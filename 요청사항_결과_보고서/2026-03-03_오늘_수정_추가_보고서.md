# 2026년 3월 3일 작업 보고서

오늘 반영한 **캔버스 Firestore 읽기 절감** 관련 수정 사항을 정리한 보고서입니다.

---

## 1. 진행한 작업 요약

### 1.1 디버그 모드에서 캔버스 초기 로드량 축소 (테스트 시 Read 비용 절감)

- **배경**: 테스트만 진행했을 때도 Firestore 일일 읽기/쓰기 한도를 초과했던 상황이 있었음. 리얼타임(RTDB) 없이 Firestore만 사용하는 구간·테스트 반복 시 Read가 많이 발생함.
- **목적**: 개발·테스트 시 캔버스 진입 시 Firestore Read를 줄여 일일 한도 소모를 완화.
- **구현**:
  - **CanvasController** `loadRoom()` 내부에서, **디버그 모드(`kDebugMode`)** 일 때만 strokes / texts / media 초기 로드 limit을 기존보다 작게 적용.
  - **리얼 빌드(Release)** 에서는 기존과 동일한 limit 유지 → 사용자 경험 변경 없음.

| 항목 | 기존 (공통) | 디버그 모드 시 |
|------|-------------|----------------|
| strokes 초기 로드 | 120건 | 40건 |
| texts 초기 로드 | 80건 | 30건 |
| media 초기 로드 | 60건 | 30건 |
| shapes | 전체 로드 (변경 없음) | 동일 |

- **효과**: 캔버스 진입 1회당 Read 수가 디버그 빌드에서 약 260건 수준 → 약 100건 수준으로 감소 (strokes+texts+media 기준). 테스트·핫 리로드 반복 시 일일 한도 초과 위험 완화.

**수정 파일**: `ink_talk/lib/screens/canvas/canvas_controller.dart`

---

## 2. 수정된 코드 위치

- **import**: `package:flutter/foundation.dart` 에서 `kDebugMode` 사용.
- **로직**: `loadRoom()` 내부, Firestore 1회 로드 직전에  
  `reduceReads = kDebugMode` 로 두고,  
  `getStrokesPage(roomId, limit: strokeLimit)`,  
  `getTextsPage(roomId, limit: textLimit)`,  
  `getMediaPage(roomId, limit: mediaLimit)` 에 위 표의 limit 적용.

---

## 3. 참고 사항

- **리얼타임(RTDB)**: 현재 구조에서는 캔버스 실시간 이벤트는 RTDB로 처리하고, Firestore는 1회 로드·추가 페이지 로드용으로 사용. 디버그 모드 limit 축소는 “리얼타임 없을 때”가 아니라 **테스트 시 비용 절감** 목적.
- **추가 절감**: 한도가 여전히 부족하면 RoomProvider/FriendProvider 등 Firestore 스트림 구독 빈도·캐시 정책, 또는 디버그 전용 “스트림 비활성화” 옵션 등을 검토할 수 있음.
